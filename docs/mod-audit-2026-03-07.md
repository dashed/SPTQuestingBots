# SPTQuestingBots Audit

Date: 2026-03-07

Status note: this report captures the pre-remediation audit state from 2026-03-07.
The issues documented below were addressed in the follow-up code and test changes
made afterward on the same date.

## Scope

This review covered the client plugin, server plugin, config contract, spawning pipeline, and the existing automated test suite.

Validation performed:

- Ran `~/.dotnet/dotnet test SPTQuestingBots.sln -v minimal`
- Result: `4538/4538` tests passed
- Read key runtime entry points, spawning code, routing/config code, and cleanup paths

## Findings

### High: bot generation can hang indefinitely on persistent `NullReferenceException`

Files:

- `src/SPTQuestingBots.Client/Components/Spawning/BotGenerator.cs:480-560`
- `src/SPTQuestingBots.Client/Patches/Spawning/GameStartPatch.cs:99-115`

Why this matters:

- `GenerateBotGroup()` loops until `botSpawnInfo != null`.
- A `NullReferenceException` is treated as transient and retried forever with no retry cap, timeout, cancellation path, or escalation.
- `GameStartPatch.waitForBotGenerators()` blocks progression while `BotGenerator.RemainingBotGenerators > 0`.

Impact:

- If upstream bot creation enters a stable failure mode, raid start can stall forever instead of failing fast and recovering.

Recommendation:

- Add a bounded retry budget.
- Add backoff and structured failure reporting.
- Ensure generator state is marked failed so the game-start wait loop can exit cleanly.

### High: `RaidHelpers.ForcePScavs` leaks globally if PScav generation throws

Files:

- `src/SPTQuestingBots.Client/Components/Spawning/PScavGenerator.cs:80-110`
- `src/SPTQuestingBots.Client/Helpers/RaidHelpers.cs:39-60`

Why this matters:

- `PScavGenerator.GenerateBotGroupTask()` sets `RaidHelpers.ForcePScavs = true`, awaits `GenerateBotGroup(...)`, then resets the flag.
- The reset is not inside a `finally`.
- `RaidHelpers.ShouldSpawnPScavByChance()` returns that global flag whenever QuestingBots PScav spawning is enabled.

Impact:

- Any exception path that exits `GenerateBotGroupTask()` before line 103 leaves the process-wide flag stuck `true`.
- After that, later spawn decisions can be polluted for the rest of the raid.

Recommendation:

- Wrap the force-toggle in `try/finally`.
- Prefer passing explicit spawn intent through method parameters instead of a mutable static flag.

### Medium: unsupported SPT versions are detected only after patches are already enabled

Files:

- `src/SPTQuestingBots.Client/QuestingBotsPlugin.cs:20-150`
- `src/SPTQuestingBots.Client/Patches/TarkovInitPatch.cs:27-65`

Why this matters:

- `QuestingBotsPlugin.Awake()` enables Harmony patches immediately.
- Version validation only runs later inside `TarkovInitPatch.PatchPostfix()`.
- On failure, the code only appends to `Chainloader.DependencyErrors`; it does not unpatch or abort the already-installed hooks.

Impact:

- On an unsupported SPT build, the mod can still mutate runtime behavior before the incompatibility is surfaced.
- That makes upgrade failures harder to diagnose and can leave the game in a partially-patched state.

Recommendation:

- Perform the SPT version preflight before any `Enable()` calls.
- Abort plugin initialization immediately on mismatch.

### Medium: config array validation is too weak for the math that consumes it

Files:

- `src/SPTQuestingBots.Server/QuestingBotsServerPlugin.cs:217-303`
- `src/SPTQuestingBots.Client/Controllers/ConfigController.cs:304-374`

Why this matters:

- Server-side validation only checks:
  - array is non-empty
  - each row has exactly 2 columns
  - left column is integral for selected arrays
- Client math assumes much more:
  - `InterpolateForFirstCol()` assumes ordered X values
  - `GetValueFromTotalChanceFraction()` assumes sane, non-negative weights

Impact:

- A malformed user config can pass startup validation but still produce inverted interpolation, broken weighted selection, or impossible percentages.
- This is especially relevant because `config.json` is intended to be user-edited.

Recommendation:

- Validate ascending left-column ordering.
- Validate legal ranges for fraction/chance arrays.
- Reject negative weights.
- Add per-array semantic checks where sums or bounds matter.

### Low: `AdjustPScavChance` is semantically drifted and appears unused

Files:

- `src/SPTQuestingBots.Client/Controllers/ConfigController.cs:67-74`
- `src/SPTQuestingBots.Server/Routers/QuestingBotsDynamicRouter.cs:59-82`
- `src/SPTQuestingBots.Server/QuestingBotsServerPlugin.cs:94-103`

Observation:

- The client helper sends the interpolated second-column value from `chance_vs_time_remaining_fraction`.
- The server route interprets the URL segment as a multiplier of `BasePScavConversionChance`.
- With the shipped config values (`0`, `10`, `20`, `50`), invoking that route would drive `ChanceAssaultScavHasPlayerScavName` far above normal percentage bounds.
- A repo-wide search found no current call sites for `ConfigController.AdjustPScavChance(...)`.

Impact:

- This does not look like an active production path today, but it is misleading and unsafe if revived later.

Recommendation:

- Remove the dead path, or
- Rename and rework both sides to one explicit contract (`chancePercent` or `multiplier`) and add behavior tests for the applied value.

## Strengths

- The repo is unusually well-tested for a game mod. The client test project contains broad coverage of config, state machines, ECS logic, regressions, concurrency, readme validation, and lifecycle cleanup.
- Client/server responsibilities are separated cleanly enough to audit.
- Static-state cleanup at raid end is much better than average for Unity/BepInEx mods.
- Reflection-heavy areas are at least partially guarded by startup validation and regression tests.

## Residual Risks

- The mod still depends on many obfuscated EFT/SPT internals and Harmony patch targets. That is manageable, but every SPT/EFT update remains a compatibility event.
- Several server files explicitly suppress obsolete `ConfigServer` warnings with a note that SPT 4.2 migration is still pending. That is a real maintenance hotspot even if it is not a current defect.
- I did not run live in-game raids in an actual SPT install, so this audit is based on code review plus automated tests, not gameplay verification.

## Overall Assessment

The codebase is in better shape than most SPT mods: documentation is strong, tests are extensive, and the architecture is understandable. The main weaknesses are concentrated in failure handling around spawning and in a few compatibility/config edges. The two spawn-pipeline exception-safety issues are the most important fixes if the goal is to harden raid startup reliability.
